# Schema Design — Документная модель MongoDB

## Обзор предметной области

Система управления задачами с пользователями и целями. Три основные сущности:
- **Users** — пользователи системы (хранятся в PostgreSQL)
- **Goals** — цели пользователей (хранятся в PostgreSQL)
- **Tasks** — задачи, привязанные к целям и исполнителям (хранятся в MongoDB)

---

## Коллекция `tasks`

### Структура документа

```json
{
  "_id": ObjectId("69e53a9e534447850212f89e"),
  "goal_id": "goal_123",
  "assignee_id": "user_456",
  "status": "pending",
  "text": "Пример текста задачи"
}
```

### Поля

| Поле          | Тип    | Обязательное | Ограничения                                      |
|---------------|--------|:------------:|--------------------------------------------------|
| `_id`         | ObjectId | авто       | Генерируется MongoDB                             |
| `goal_id`     | string | Да           | Не пустая строка, макс. 128 символов             |
| `assignee_id` | string | Да           | Не пустая строка, макс. 128 символов             |
| `status`      | string | Да           | Enum: pending, in_progress, done, cancelled      |
| `text`        | string | Да           | Не пустая строка, макс. 2000 символов            |

---

## Обоснование выбора: Embedded vs References

`goal_id` и `assignee_id` хранятся как строки-ссылки т.к хранилища разделены. Плюс не обязательно будет связь один к одному если цель

Использовать стоит embedded если цели существовали в mongodb и мы хотели бы за один запрос получать полностью информацию с целью и задачей.
